目录树与物理约束
📁 react-workflow / 目录树
react-workflow/ ├── .claude-plugin/ ← 唯一必需子目录 │ └── plugin.json ← 清单文件 ├── commands/ ← 斜杠命令 ├── agents/ ← 子智能体 ├── skills/ ← Skills ├── hooks/ ← Hooks 配置 ├── .mcp.json ← MCP 配置 └── README.md
🔒 物理约束
• 只有 plugin.json 必须放在 .claude-plugin/ 子目录内
• 其他组件直接位于根目录
✅ 设计巧思
Plugin 根目录本身就是

plugin.json · Plugin 的身份证
📋 plugin.json
{ "name": "react-workflow", "version": "1.2.0", "description": "Complete React workflow", "author": "YourName", "repository": "github.com/yourname/...", "license": "MIT", "keywords": ["react", "workflow"] }
结构与 package.json 高度相似
📋 字段规范
字段约束说明
name✅ 必需唯一标识符
version✅ 必需语义化版本号
description✅ 必需一句话简介
author⚪ 可选作者署名
repository⚪ 可选仓库地址
license⚪ 可选开源协议
keywords⚪ 可选检索关键词
[ 命名即命名空间 ]
name · 决定命名空间
✅ 推荐命名
react-workflow｜一看就知道是 React 工作流
pr-reviewer｜直接表达 PR 审查能力
db-tools｜数据库工具集，简短不冲突
❌ 避免命名
my-plugin｜过于宽泛，没有信息量
tools｜极易与他人重名冲突
v2｜无意义，看不出做什么
🔑 安装后变成 plugin-name:component-name，名字就是命名空间
[ 组件格式 · 上 ]
组件① 命令 + ② 子智能体

命名空间 · 多 Plugin 共存

设计原则 ③ · ④
最小权限 · 文档必需
🛡️ 最小权限
✅ 推荐 ｜只读三件套
Read, Grep, Glob
❌ 禁止 ｜过权·高风险
Write, Bash
🔑 权限越大 · 信任越贵
📄 文档是必需品
README.md 必须包含：
一行安装命令（可复制）
完整功能清单（Commands / Agents / Skills）
配置指南（环境变量、MCP）
清晰的更新日志
💥 没有文档 = 没有价值

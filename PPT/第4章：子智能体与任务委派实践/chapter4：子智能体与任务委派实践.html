上下文窗口的信噪比

Claude 的上下文约 20 万 Token，但 "大" 不等于 "好用" — 关键看信噪比。
【原始噪声】
200 行
堆栈、警告、堆栈、警告 ……
几百行测试输出全量倒入主对话
【真正的信号】
1 行
3 个测试失败 / 47 个通过
真正需要的，往往只是一句结论 — 剩下的全是噪声
噪声稀释信息 · 注意力被分散 · Claude 失焦

子智能体的本质

[ DEFINITION ]

一个具备 独立上下文窗口、受限工具权限 与 明确任务范围 的 Claude 实例。
📚
独立上下文
中间过程严格保留在自身上下文，绝不回流主对话
🔒
受限工具权限
工具白名单从物理层面禁止越权操作
➕
明确任务范围
仅向主智能体返回结论，而非过程中的所有细节

三大核心价值

对应三条软件工程原则
1
隔离
独立上下文窗口 — 文件内容、命令输出、中间推理严格保留在自身空间
Bulkhead 舱壁模式
2
约束
工具权限白名单 — 物理边界，而不是 Prompt 中的 "请不要" 建议
Least Privilege 最小权限
3
复用
Markdown 定义 — 团队共享、项目迁移，新员工入职即可直接使用
Componentization 组件化

技术实现 · 子进程独立窗口

[ CHILD PROCESS · CONTEXT ]
1  System Prompt
2  项目级 CLAUDE.md

① 启动全新子进程
② 子进程内独立执行：读文件、运行命令、分析结果
③ 仅回传 结果摘要 至主进程

YAML 前置元数据三维度

--- name: code-reviewer # 身份维度 description: 审查代码质量、安全漏洞与性能问题 # 身份维度 tools: [Read, Grep, Glob] # 权限维度 model: sonnet # 执行维度 ---
身份维度
name description
它是谁、何时被触发，决定主智能体的路由匹配
权限维度
tools permissionMode
工具白名单 — 最小权限原则，物理层禁止越权
执行维度
model hooks
在哪里运行 / 模型选择 / 期间发生什么

两种调用方式

主对话最终收到的，永远是经过提炼的结论 — 而非原始噪声
💡 ① 自动委派
用户：帮我审查 src/payment/ 目录的代码变更
Claude：好的，委派给 code-reviewer
→ 启动子智能体 ...
基于 description 语义相似度，自动匹配合适的子智能体
✅ ② 显式请求
用户：用 @code-reviewer 审查这段代码

→ 直达指定 ...
用户明确指定子智能体名称，绕过语义匹配
→ 主对话只看到精炼结论，所有细节留在子上下文

模式① 只读型

[ READ-ONLY OBSERVER ]
只读不写 — 赋予观察权，剥夺修改权。审计员可以查阅账本，但绝不允许篡改账本。
[ ALLOWED TOOLS ]
◎  Read   读取文件内容
◎  Grep   模式搜索
◎  Glob   文件名匹配
[ DENIED · 物理边界 ]
✖  Edit   修改文件
✖  Write   写入新文件
✖  Bash   执行命令
→ 最小权限原则的完美体现 · 即使幻觉爆发也无法越界

[ NOISE FILTER ]

模式④ 流水线型

[ CHAIN OF RESPONSIBILITY ]
具有明确阶段依赖的任务流 — 每个阶段的输出，是下一阶段的输入。
STAGE 01
定位

输出：根因定位
STAGE 02
修复

输出：修复方案
STAGE 03

权限：只读+Bash
输出：验证结果
STAGE 04

交接契约（Handoff Contract）
每阶段的输出格式必须严格匹配下一阶段的输入要求 — 类比 Unix 管道： cat | grep | sort | uniq

子智能体与 Skills 协作

只有两种原子模式
[ DIRECTION · A ]
子智能体 → 预加载 Skill
子智能体 ⊃ Skill
name: bug-fixer skills: - secure-coding
子智能体是 执行者（角色驱动）
Skill 是它手中的 操作手册
场景：流水线长期角色 · 多轮交互
[ DIRECTION · B ]
Skill → 派生子智能体
Skill ⊃ 子智能体
name: codebase-check context: fork agent: general-purpose
Skill 是 调度者（流程驱动）
子智能体只是 执行容器
场景：单次重型任务 · 严格上下文隔离
判断标准：谁是主角？ · 角色驱动选 A · 流程驱动选 B

实战注意事项

这些细节决定子智能体能否在生产环境稳定工作
[ TIP · 01 ]
CLAUDE.md 继承关系
所有子智能体自动加载项目级 CLAUDE.md
优先级：子智能体 > 项目级 > 用户级
[ TIP · 02 ]
报文传输模式
子智能体之间无法直接通信
主智能体必须显式提取并转发结论
[ TIP · 03 ]
中断恢复机制
中断会丢失内存中的中间产物
耗时任务务必持久化到文件
[ TIP · 04 ]
嵌套深度控制
嵌套层级建议 ≤ 2 层
超过 3 层 = 任务分解需要重新设计
⚠️
子智能体不是银弹 — 简单任务直接做，不要过度设计

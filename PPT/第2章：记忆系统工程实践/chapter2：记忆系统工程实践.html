FRAMEWORK · 三问框架

精选内容的黄金标准

🔵 WHY
为什么这样做 · 举一反三 · 推理能力
📌 选择 Fastify 而非 Express，因其 schema-based validation 符合"类型安全优先"原则，有助于 Claude 推断类似场景的选择。
🟡 WHAT
做什么 / 不做什么 · 行为边界 · 明确规范
📌 必须使用 pnpm，严禁使用 npm 或 yarn。此类规则无需 WHY，一句话即可。
🟢 HOW
如何执行 · 标准作业程序 · SOP
📌 pnpm db:migrate 执行迁移，pnpm db:seed 填充测试数据，顺序不可颠倒。
💡 大多数规则只需 WHAT 一句话，只有容易误解的才补充 WHY

PYTHON 数据项目

数据处理统一约定

❌
容易出错的反面模式
# ❌ inplace=True 破坏数据流
df.drop_duplicates(inplace=True)
df.fillna(0, inplace=True)

# ❌ 混用 None / np.nan / pd.NA
df['col'] = None
df['col'] = np.nan
df['col'] = pd.NA
⚠️ 后果：数据状态不可追踪、调试困难
✅
统一正确约定
✔️ 缺失值统一：始终使用 pd.NA
✔️ 禁止 inplace：所有操作返回新 DataFrame
✔️ 日期类型统一：强制 datetime64[ns]
✔️ 低基变量优化：分类变量转 category
🔧 推荐工作流
uv sync → 同步依赖
uv run pytest → 运行单元测试
uv run jupyter lab → 启动交互分析
GOLDEN RULE · 黄金检验标准

内容去留的终极判断

✨
"如果删去某条规则后，Claude 依然能做出正确行为，
那么这条规则不该出现在 CLAUDE.md 中。"
—— 用于裁剪冗余规则的核心判据
✅
必要性
Claude 必须知晓且无法自行推断
规则缺失将导致行为偏差
◎
可操作性
具体、明确、可立即执行
无模糊表述（如"尽量""通常"）
⚠️
脆弱性测试
删除后会直接导致 Claude 犯错
是"临界点规则"
💡 任一维度不满足 → 该条规则应删除或重构

PreToolUse

最强大
工具决定调用后 · 实际执行前触发
✅
allow
绕过权限系统直接执行
❌
deny
阻止执行并说明原因
✏️
updatedInput · 最强大
修改输入参数后执行
修改输入：在不中断操作的前提下，静默添加安全参数

updatedInput 实战示例

hookSpecificOutput 格式
{ "hookSpecificOutput": { "hookEventName": "PreToolUse", "permissionDecision": "allow", "updatedInput": { "command": "rm -rf /tmp/test --dry-run" } } }
效果：
rm -rf /tmp/test
↓ 静默改为
rm -rf /tmp/test --dry-run

工具调用事件 · 其余4个

• PostToolUse
执行后
工具成功执行后触发。无法撤销操作，但可通过 additionalContext 向 Claude 反馈信息
典型用途：lint 结果反馈 · 自动格式化
• PostToolUseFailure
工具执行失败后触发
典型用途：错误告警 · 纠正性反馈
• PermissionRequest
权限对话框弹出前触发
vs PreToolUse：仅在需要用户手动确认时激活
以编程方式自动批准或拒绝权限
• UserPromptSubmit
用户提交输入后、Claude 开始处理前触发
典型用途：输入预处理 · 自动附加 Git 分支信息

子智能体事件

• SubagentStart
子智能体启动时触发。不能阻止启动，但可通过 additionalContext 注入关键上下文
示例：启动 code-reviewer 时自动加载团队编码规范
• SubagentStop
任务完成后触发。可以拦截，强制子智能体继续工作直至满足质量标准
transcript_path agent_transcript_path

完成事件

• Stop
质量门控核心
Claude 完成整轮响应时触发。检测到不达标时 block，强制继续修正
decision: "block" reason: "测试失败，请修复后再停止"
• Notification
系统发送通知时触发，可区分：
permission_prompt idle_prompt auth_success
集成 Slack / 钉钉 · 本地桌面弹窗

type 协议规范

type 说明 典型场景
command 执行本地 Shell 命令，JSON 通过 stdin 传入，exit code 控制行为（0=通过，2=阻断） 拦截危险命令、跑 lint、格式化
http POST JSON 到指定 URL，响应同 command 格式 Slack/飞书通知、审计日志上报
mcp_tool 直接调用已连接的 MCP Server 的工具 与已有 MCP 工具联动
prompt 将 prompt 发给轻量 Claude 模型做单轮判断 语义检查（如"这段代码是否泄露密钥"）
agent 启动子 Agent，可使用 Read/Grep 等工具多步验证 复杂代码审查、多条件校验
💡
设计原则：所有类型共享统一响应格式（permissionDecision, reason, updatedInput），确保 Hooks 层逻辑一致性。

退出码 · 意图信号

✅
0
成功 · 放行
脚本正常完成，操作继续执行
❌
2
有意阻止 · 关键！
操作被主动拦截，严格区分于脚本故障
⚠️
1/3+
脚本异常 · 透传错误
脚本自身出错，不应影响正常工作流

设计哲学

☕ 烟雾报警器类比
退出码 2 = 真实火警
报警器检测到烟雾 → 有意触发 → 人员疏散
退出码 1 = 报警器故障
报警器自身断电/坏掉 → 不应禁止人员进出大楼
核心原则
脚本故障 不应 阻碍正常工作流运行

hookSpecificOutput

PreToolUse 与 Claude 通信的专用协议
2025 年升级为嵌套对象格式（推荐）
✅
allow
绕过权限系统直接执行
❌
deny
阻止执行，附带原因说明
⚪
ask
我不确定，请由人工决定
📥
additionalContext
适用于所有事件类型，内容注入 Claude 上下文，构建高效反馈闭环
完整协议格式（推荐嵌套格式）
stdout 输出格式
{ "hookSpecificOutput": { "hookEventName": "PreToolUse", "permissionDecision": "deny", "permissionDecisionReason": "禁止提交 .env 文件", "additionalContext": "检测到敏感文件..." } }
allow + updatedInput（修改参数执行）
{ "hookSpecificOutput": { "hookEventName": "PreToolUse", "permissionDecision": "allow", "updatedInput": { "command": "cmd --dry-run" } } }

第一道防线：脚本核心设计
hooks/block-dangerous.sh
#!/bin/bash INPUT=$(cat) # 从 stdin 读取 Claude 请求 # jq 解析（避免脆弱的字符串匹配） CMD=$(echo "$INPUT" | jq -r '.tool_input.command // ""') if echo "$CMD" | grep -qE 'rm -rf /'; then # 生成 deny 响应并输出到 stderr jq -n '{ hookSpecificOutput: { hookEventName: "PreToolUse", permissionDecision: "deny", permissionDecisionReason: "危险命令已拦截" } }' >&2 exit 2 # ← 关键：退出码2 = 强制阻断 fi
✅ stderr 输出调试
✅ jq 安全解析
✅ 明确退出码语义

第二道防线 敏感文件保护

matcher 配置 – 精准触发
"matcher": "Write|Edit"
• 仅在文件写入或编辑时触发
🛡️ 保护范围三类
受保护文件名　.env　credentials.json　.netrc
受保护扩展名　.pem　.key　.p12　.pfx
受保护目录　.git/　.ssh/　/etc/　~/.aws/
🔔
如果这道防线当天就位，萌新的悲剧本可完全避免
核心检测逻辑
hooks/protect-sensitive.sh
FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // ""') BASENAME=$(basename "$FILE") EXT="${BASENAME##*.}" # 检查受保护文件名 PROTECTED_NAMES=(.env credentials.json) for name in "${PROTECTED_NAMES[@]}"; do [[ "$BASENAME" == "$name" ]] && deny "敏感文件禁止写入：$BASENAME" done # 检查受保护扩展名 PROTECTED_EXTS=(pem key p12) for ext in "${PROTECTED_EXTS[@]}"; do [[ "$EXT" == "$ext" ]] && deny "密钥文件禁止写入：.$EXT" done

PostToolUse Hook 实现
hooks/auto-format.sh
FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // ""') EXT="${FILE##*.}" case "$EXT" in ts|js|tsx|jsx) command -v prettier >/dev/null && prettier --write "$FILE" >/dev/null ;; py) command -v black >/dev/null && black "$FILE" >/dev/null ;; esac exit 0 # 非阻断：即使无工具也允许流程继续

Hook 脚本关键实现
hooks/lint-feedback.sh
FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // ""') # 仅检查 JS/TS 文件 if [[ "$FILE" =~ \.(js|ts)$ ]]; then LINT_RESULT=$(eslint "$FILE" --format compact 2>&1) # 错误注入 Claude 上下文 if [ -n "$LINT_RESULT" ]; then jq -n '{ additionalContext: "Lint 错误:\n'$LINT_RESULT'" }' | cat >&2 fi fi exit 0

Stop Hook · 质量门控

Claude 宣称"完成"时触发，测试失败则阻止会话结束
hooks/quality-gate.sh
# 关键：防止死循环 if [[ "${STOP_HOOK_ACTIVE}" == "true" ]]; then # 已在 Stop Hook 中，放行避免无限循环 exit 0 fi # 运行测试套件 TEST_RESULT=$(npm test 2>&1) TEST_EXIT=$? if [[ $TEST_EXIT -ne 0 ]]; then # 测试失败 → 阻止停止，强制继续修复 jq -n '{ decision: "block", reason: "测试失败，请修复后再停止" }' >&2 exit 2 fi
关键机制
⚠️ STOP_HOOK_ACTIVE
Stop Hook 执行失败 → 系统再次触发
标志置为 true → 脚本检测后放行
打破无限修复循环
↩️ 递归类比
就像递归函数的 终止条件
没有它，Claude 会陷入：
修复 → 测试失败 → 再修复 → ...
永远无法自行脱困
测试失败 → block → Claude 修复
再次完成 → 测试通过 → 允许停止

异步 Hooks

2026 年初
"async": true # Hook 在后台非阻塞运行
Claude 无需等待 Hook 完成即可继续工作
⚠️ 两个关键限制
→ 仅 command 类型支持异步
→ 无法阻止当前操作（只读/通知用途）
适用场景
日志记录 异步通知 监控上报
🔧 调试三板斧
①

echo "Debug: $VAR" >&2

②
手动构造输入直接测试

无需运行完整会话即可测试脚本

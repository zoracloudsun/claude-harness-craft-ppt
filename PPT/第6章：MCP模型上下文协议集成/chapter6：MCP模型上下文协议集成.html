⚠️ 关键安全原则

数据库连接必须只读
🚫 危险：读写账号
Claude 理论上能执行：
INSERT INTO ... UPDATE table SET ... DROP TABLE users
→ 误操作 = 数据灾难
🛡️ 安全：只读账号
仅允许查询动作：
SELECT col FROM ... WHERE / GROUP BY GRANT SELECT ONLY
→ 最小权限原则
📌 Read-Only User = 最基本防护

[ 安全机制 ]

安全的三重防御

扩展能力边界 · 也守住信任底线
🛡️ Layer 1
交互式审批
• 首次连接服务器时，用户交互确认
• 建立信任基石
$ claude mcp add
→ "你信任此服务器吗？[y/N]"
🛡️ Layer 2
工具级权限
• 细粒度控制每个工具的调用权限
• 敏感操作需二次确认
tool: postgres.query
→ "允许执行 SQL? [y/N]"
🛡️ Layer 3
OAuth 2.0
• 动态认证替代静态密钥
• 令牌可撤销 · 范围最小化
OAuth flow → token
→ 短时效 / 自动刷新 / 可撤销
[ 风险与防护 ]

三类安全风险与防护

研究人员揭示 · 防护可执行
🚫
提示注入攻击
恶意内容通过 MCP 数据流注入上下文，诱导非预期操作
✅ 防护策略
仅安装官方与知名社区服务器，警惕未审计来源
⚠️
工具权限滥用
多工具组合调用 → 读 + 写 + 网络 = 敏感数据泄露
✅ 防护策略
数据库只读账号 · 最小权限原则 · 禁用危险操作
👤
冒名顶替
恶意服务器伪装合法工具，骗取密钥或执行有害命令
✅ 防护策略
API 密钥 via 环境变量 · 不写入代码 · 不外传配置

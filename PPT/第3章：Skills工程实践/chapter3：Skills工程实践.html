三步黄金策略
01
写什么
动宾短语描述核心功能，精确说明 Skill 能做什么
• "分析 git 变更"
• "生成测试报告"
• "审查代码质量"
02
何时触发
列出自然语言触发词，覆盖用户可能的表述方式
• "触发词：提交、commit"
• "当用户需要 git 操作"
• "用于 PR 合并前"
03
不适用于
明确排除边界，防止误触发，提高精准度
• "不适用于代码审查"
• "仅限 git 操作场景"
• "不处理部署流程"
黄金公式：[功能动词 + 对象] + [触发场景] + [边界排除] = 完美 description
控制在 40–80 tokens 之间，语义密度最高
触发失败 vs 触发过度
触发失败 / Under-trigger
56%
由 description 过于模糊导致
"用户说了提交代码，Skill 没有激活，Claude 徒手操作 git"
→ 用户不知道 Skill 存在
→ 丢失标准化的提交格式
→ 需要手动重复操作
触发过度 / Over-trigger
噪音
由 description 过于细化导致
"用户只是问了一句 git 是什么，Skill 就开始执行提交流程"
→ 打断用户正常对话
→ 浪费大量 Token
→ 用户体验极差
验收标准（黄金区间）
● 意图明确时 100% 触发
○ 闲聊/问答时 0% 触发
✧ 模糊意图时询问确认

参考型 Skill
Reference Skill

allowed-tools 四种模板
📋
审计型
allowed-tools: - Read - Glob
只读权限，代码审查、文档分析，零副作用
✏️
生成型
allowed-tools: - Write - Edit - Read
读取后生成或修改文件，代码生成、重构
🔍
分析型
allowed-tools: - Read - Bash(git log:*) - Bash(git diff:*)
读取 + 有限 Bash，分析代码历史和变更
⚙
执行型
allowed-tools: - Bash(git add:*) - Bash(git commit:*) - Read
精确 Bash 前缀，最小权限执行 git 操作
※ Bash(git status:*) — 前缀匹配：允许 git status 及所有参数，但禁止 git push 等危险操作
Bash 精细化权限控制
✓ 精准前缀匹配
Bash(git status:*)
✓ 允许查看状态，禁止所有写操作
Bash(git add:*)
✓ 允许暂存文件，禁止 push/reset
Bash(git commit:*)
✓ 仅允许提交，精确控制
⚠ 危险：Bash(*)
Bash(*) = 允许执行任意 Shell 命令
※ 提示注入攻击窃取密钥
※ 删除生产数据库
※ 植入后门脚本
※ 向外网发送敏感信息
遵循最小权限原则：只授予完成任务所需的最小权限

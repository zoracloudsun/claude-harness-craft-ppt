import asyncio from claude_agent_sdk import query, ClaudeAgentOptions async def main(): options = ClaudeAgentOptions( max_turns=3, allowed_tools=["Read", "Grep"], system_prompt="You are a code reviewer.", ) async for msg in query( prompt="分析 src/api 模块", options=options ): if msg.type == "assistant": print(msg.content, end="", flush=True) elif msg.type == "result": print(f"\n成本: ${msg.cost} · 耗时: {msg.duration}s") asyncio.run(main())

ASSISTANT
Claude 的每一轮"开口说话"
content 是一个数组 —— 混合 text（思考输出）与 tool_use（工具调用请求）
📋 消息结构示例
{ "type": "assistant", "content": [ { "type": "text", "text": "我先并行读两个文件..." }, { "type": "tool_use", "id": "toolu_01A", "name": "Read", "input": { "path": "a.ts" } }, { "type": "tool_use", "id": "toolu_01B", "name": "Read", "input": { "path": "b.ts" } } ] }
≡ text block
Claude 的思考、说明、自然语言输出 —— 给人看的
🔧 tool_use block
工具调用请求 —— 给宿主程序执行的指令
» 并发执行
单条消息可含多个 tool_use —— 这意味着并发执行
USER · TOOL_RESULT
id 严格配对 · SDK 自动维护
ASSISTANT → tool_use
Claude 发起工具调用
"id": "toolu_01ABC", "name": "Read"
USER → tool_result
宿主回喂执行结果
"tool_use_id": "toolu_01ABC", "content": "file contents..."
↔ 自动配对 tool_use_id ≡ id
📝 消息体示例
{ "type": "user", "content": [ { "type": "tool_result", "tool_use_id": "toolu_01ABC", // ← 严格对应上一条 assistant 的 tool_use.id "content": "export function fetch() { ... }", "is_error": false } ] }
RESULT

自定义工具 · @tool 装饰器
把业务函数注册成 Claude 可调用的工具，进程内运行、零 IPC 开销
🐍 工具定义 (tools.py)
# 1. 装饰器声明单个工具 @tool( name="query_db", description="按订单号查询数据库", parameters={"order_id": "str"} ) async def query_db(order_id: str): return await db.find(order_id) # 2. 打包成进程内 MCP 服务器 from claude_agent_sdk import create_sdk_mcp_server biz = create_sdk_mcp_server( name="biz", tools=[query_db, call_api, notify] ) # 3. 注册进 options options = ClaudeAgentOptions(mcp_servers={"biz": biz})
📦 @tool 扁平注册
一行声明 name / description / parameters，函数体即工具实现
➕ create_sdk_mcp_server
多个 @tool 打包成一个 MCP 服务器，统一挂载
⚡ 进程内运行
跑在当前进程，无进程间通信开销，毫秒级响应
→ Agent → 进程内 MCP 服务器 → [query_db, call_api, notify]
大佬提醒
工具设计 · 三大原则
工具是 Claude 的能力延伸，每一项都要谨慎对待
🎯
原则 01：单一职责
每个工具只做一件事。不要设计"万能查询" —— 边界越宽，误用风险越高
# bad do_anything(action, target) # good get_order(order_id) cancel_order(order_id)
📋
原则 02：描述清晰
description 是写给 Claude 看的使用指南。明确 "何时可用" 与 "何时禁用"
@tool(name="refund", description=""" 仅订单状态=已完成时退款 禁止：处理中、争议中订单 """)
🛡️
原则 03：零信任验证
永远校验输入。Claude 可能传入任何字符串，不要假设它正确
async def refund(order_id): if not ORDER_RE.match(order_id): raise ValueError(...) if not await is_finished(order_id): return {"error": "..."}
Hooks 演进 · 配置 → 编程
第5章的 Shell Hooks 是 JSON 配置 + 脚本；SDK 把它升级为原生编程接口
维度 Shell Hooks (第5章) SDK Hooks (升级版)
定义方式 JSON 配置 + Shell 脚本 原生 Python / TypeScript 函数
执行环境 独立子进程 主进程内闭包
数据交换 stdin / stdout 序列化 JSON 直接函数参数 + 返回值
异步支持 ❌ 不支持，阻塞 stdio ✅ 原生 async / await
调试体验 ⚠️ 只能 print 打日志 ✅ 断点 · 单步 · 完整 traceback
适用场景 CLI 用户的本地配置 / 团队规则 嵌入式应用 / 服务端集成
🔁 一句话总结：嵌入式应用用 SDK Hooks，纯 CLI 工作流才用 Shell Hooks

结构化输出 · JSON Schema 强约束
声明 output_format，Claude 输出自动按 schema 验证，失败自动重试
🐍 Python · Pydantic
from pydantic import BaseModel class OrderInfo(BaseModel): order_id: str amount: float status: str options = ClaudeAgentOptions( output_format=OrderInfo, # ← 自动校验 · 失败重试 ) # result.parsed → OrderInfo
📜 TypeScript · Zod
import { z } from "zod"; const OrderInfo = z.object({ order_id: z.string(), amount: z.number(), status: z.string(), }); const options = { outputFormat: OrderInfo, }; // result.parsed → z.infer<typeof OrderInfo>

选型决策
SDK vs CLI · 场景对照
经验法则 · 一行 Shell 描述用 CLI；写 if/else 或循环用 SDK
场景 推荐 原因
CI/CD 单步任务 CLI 单行命令直接调用，无需写代码
Shell 脚本自动化 CLI 管道组合、易于嵌入 bash/zsh
构建 Web / 桌面 AI 应用 SDK 需要 HTTP/WebSocket、UI 状态管理
需要自定义工具 SDK @tool 装饰器 + 进程内 MCP
多轮对话会话 SDK 维护 session、上下文状态
精细控制消息流 SDK 逐条 message 拦截、过滤、改写
高并发任务管理 SDK asyncio 并发、线程池、连接复用
🔑 选型口诀：一行 Shell 可描述 → CLI；写 if/else 或循环 → SDK
